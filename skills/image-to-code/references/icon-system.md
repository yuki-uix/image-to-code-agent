# Icon system reference

Guidance for planning code icons / UI glyphs on web pages generated by this skill (landing pages, ecommerce, dashboards). Goal: navigation, buttons, form controls, tabs, toggles, and small semantic status marks all come from one consistent code icon system — not raster crops, not generated images, not a pile of one-off hand-authored SVGs.

Boundary: a card's hero photo, a product shot, a person/object in the foreground, a lifestyle or scene image, or anything needing a transparent cutout is an **image asset** (`--asset-policy generate` or `crop`), not a UI glyph — even when its filename or alt text mentions an object that also has an icon-library equivalent (e.g. a "cart" *icon* in the header vs a "shopping bag" *product photo* in a hero banner are different things).

Decision order:

1. If the element sits in a nav bar, toolbar, button, tab, form control, filter chip, or a small interactive/status indicator, it is a code icon / UI glyph.
2. If the element is a card's main image, a product/object photo, a person/object foreground, a lifestyle or scene shot, or needs a transparent background, it is an image asset — even if its name or role overlaps with something an icon library also has a glyph for.
3. If the element is a brand mark, illustrated badge, or map pin, decide brand/decorative asset first; only treat it as a UI glyph when it actually functions as a control (back, settings, play, nav).

## 1. Allowed icon libraries

Pick exactly one for a given project/demo:

1. `@phosphor-icons/react` — broad coverage, both line and filled styles, good general default.
2. `hugeicons-react` — strong coverage for physical objects/product-adjacent glyphs.
3. `@radix-ui/react-icons` — minimal, stable, lightweight; may be thin on less common glyphs.
4. `@tabler/icons-react` — broad coverage, consistent stroke style, good for dashboards/tools.

Rules:

- If the project already uses one of these, keep using it.
- If the project already has a different primary icon library, keep that one — do not add a second.
- On a fresh React/Next/Vue output, default to `@phosphor-icons/react` or `@tabler/icons-react`.
- Do not mix Lucide, Iconify, emoji, raster icons, and hand-authored SVG paths with one of the four libraries above on the same page.

## 2. Single call-site

Establish one entry point (e.g. `UiIcon` or an `IconRegistry`); feature components call it by semantic name, never import icon libraries directly in more than one place:

```tsx
import { ArrowLeft, DotsThreeVertical, GearSix, ShoppingCart, Heart, Search, SlidersHorizontal, User, X } from "@phosphor-icons/react";

const icons = {
  back: ArrowLeft,
  more: DotsThreeVertical,
  settings: GearSix,
  cart: ShoppingCart,
  wishlist: Heart,
  search: Search,
  filter: SlidersHorizontal,
  account: User,
  close: X,
} as const;

export function UiIcon({ name, size = 20, weight = "regular", ...props }) {
  const Icon = icons[name];
  return <Icon size={size} weight={weight} aria-hidden="true" {...props} />;
}
```

For plain HTML/CSS/JS output, build one SVG sprite or an `icon(name)` helper; every glyph shares the same `viewBox`, stroke, linecap, linejoin, and size classes.

## 3. Always code-rendered

The following are always code icons, never a raster `<img>`, never `--asset-policy generate`/`crop` output:

- Navigation: back, close, menu/more, search, breadcrumb chevrons.
- Commerce controls: cart, wishlist/heart, filter, sort, quantity +/-.
- Form controls: checkbox/radio marks, dropdown chevrons, password-visibility toggle, clear-field.
- Tabs and pagination markers.
- Status: selected ring, disabled state, toggle knob, loading spinner.
- Social/share icons in a footer or share row (use the library's brand-neutral glyph, not a raster logo grab).

## 4. Coverage table template

| glyph | purpose | library name / SVG id | size | stroke/fill | container | states | aria-label |
| --- | --- | --- | --- | --- | --- | --- | --- |
| back | return to previous view | ArrowLeft / arrow-left | 20 | stroke 2 | 40x40 button | default/pressed | Back |
| cart | open cart | ShoppingCart / shopping-cart | 20 | stroke 2 | 40x40 button | empty/has-items | Cart |
| wishlist | save item | Heart / heart | 18 | stroke 2 / fill on active | 36x36 button | default/saved | Add to wishlist |
| search | open search | Search / search | 18 | stroke 2 | 40x40 button | default/active | Search |
| filter | open filters | SlidersHorizontal / filter | 18 | stroke 2 | 36x36 button | default/applied | Filters |

Do not leave the coverage table empty at delivery. If a glyph has no library match, note the local fallback SVG id and still route it through the single call-site.

## 5. Acceptance checks

- Zoom the render to 200% and check nav bar, tab bar, and card-level small icons.
- Icon style must be internally consistent: same stroke width, endcaps, corner radius, and fill strategy across the page.
- Icon and hit-area are separate concerns: the glyph can be 16-24px, but its clickable container stays at least 40x40px.
- Icon-only buttons must carry an `aria-label` or equivalent accessible name.
- Do not introduce a second icon library on a page unless the existing code already had one and this change didn't touch it; new UI always uses the primary library.
