# validate-html

Check a generated HTML file for common problems and optionally repair them.

## Usage

```
/validate-html <html-path> [--fix]
```

- `<html-path>` — path to the HTML file — **required**
- `--fix` — write a repaired version back to the same file after reporting issues

## What to do

### Step 1 — Read the file

Use the Read tool to load the HTML. Parse it mentally before evaluating.

### Step 2 — Check for issues

Go through each category below. For each problem found, note: the issue type, where it occurs (line number or component name), and the severity (error = page won't render, warning = visual degradation).

**JSX / syntax errors**
- String literals rendered as JSX children, e.g. `<div>"children"</div>` — should be `{children}`
- Bare `{PascalCase}` references used as values instead of being called as components, e.g. `{NavBar}` instead of `<NavBar />`
- Unclosed tags or mismatched braces
- `type="module"` on the Babel script tag (should be `type="text/babel"`)

**Component structure**
- Components nested inside other components' JSX instead of being siblings at the `Page` level
- A component named in JSX but never defined (will throw `ReferenceError` at runtime)
- Data arrays defined inside a render function instead of at module scope (causes re-initialization on every render)

**Visual fidelity**
- Color values that look like Tailwind defaults (`blue-600`, `gray-500`) in contexts where the original clearly used a specific hex — flag if they seem mismatched
- Placeholder or dummy text: "Lorem ipsum", "Service 1", "Card Title", "Description goes here", "TODO", "Example"
- Entire sections that appear to be missing (a hero section with no headline, a card grid with one card, etc.)

**CDN / dependencies**
- Missing `react` or `react-dom` script tags
- Missing `tailwindcss` script tag
- Google Fonts link missing when a non-system font is referenced in CSS

### Step 3 — Report findings

Output a structured summary:

```
## Validation results: <filename>

### Errors (will break rendering)
- [ComponentName] <description of issue>

### Warnings (visual issues)
- [SectionName] <description of issue>

### Passed
- JSX syntax: OK
- Component definitions: OK
- CDN dependencies: OK
- (etc.)
```

If no issues are found, say so clearly: "No issues found — the file should render correctly."

### Step 4 — Repair (if --fix)

For each error or warning, apply the minimal fix:

- `"children"` → `{children}` (string literal to JSX expression)
- `{NavBar}` → `<NavBar />` (bare reference to component call)
- Missing component definition → add a minimal stub with a visible placeholder label
- Placeholder text → keep as-is (can't infer the real content without the original image)
- Wrong CDN `type` → change to `type="text/babel"`

After repairing, write the file back using the Write tool. Then re-run the validation checks mentally to confirm no new errors were introduced.

Finish by telling the user:
- Which issues were fixed
- Which issues remain and why (e.g. "placeholder text was kept — please provide the original image to regenerate with real content")

## What this skill does NOT do

- It does not re-generate the page from scratch. Use `/image-to-code` for that.
- It does not check visual fidelity against the original screenshot — it can only check structural correctness. For visual review, open the file in a browser.
- It does not fix wrong colors or wrong layout if the JSX itself is syntactically valid — those require judgment against the original image.
